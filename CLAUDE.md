## Repository organisation

```
.
├── CLAUDE.md
├── README.md
├── config/
│   └── config.yml          ← master configuration file
└── scripts/
    ├── analysis/
    │   └── chapterX/
    │       ├── main.py
    │       ├── README.md   ← required; describes input data, output tables, and figures
    │       └── lib/
    ├── figures/
    │   └── chapterX/
    │       ├── main.py
    │       ├── lib/
    │       ├── plots/      ← generated by main.py (do not edit manually)
    │       └── figures/    ← manual Adobe files and panels (not touched by scripts)
    └── helpers/
        └── config.py
```

`scripts/` is split into two categories:

- `analysis/` — reads raw input, produces CSV tables to `cfg.output_dir / "chapterX"`
- `figures/` — reads those CSVs, writes plots to `scripts/figures/chapterX/plots/`

### Where outputs go

| Category | Reads from | Writes to |
|---|---|---|
| `analysis/chapterX/` | `cfg.input_dir` | `cfg.output_dir / "chapterX"` (external) |
| `figures/chapterX/` | `cfg.output_dir / "chapterX"` | `scripts/figures/chapterX/plots/` (in-repo) |

Analysis outputs (CSV tables) go **outside** the repository to `output_dir`.
Figure outputs (plots) go **inside** the repository, next to the script that produced them.

### figures/ subdirectory convention

Each `figures/chapterX/` directory contains two subdirectories:

- `plots/` — all plots generated by `main.py`; overwritten on each run; do not edit manually
- `figures/` — manual workspace for Adobe Illustrator files, panels, and final figures; never touched by scripts

## Chapter structure

### analysis/chapterX/

```
chapterX/
├── main.py     ← entry point; loads Config, calls lib/
├── README.md   ← required (enforced by Config); see README convention below
└── lib/        ← importable modules
```

**README.md convention** — every `analysis/chapterX/` folder must have a `README.md` that describes:
- The input data (file names, relevant sheets/columns)
- The output tables (file names, key columns)
- The figures produced by the corresponding `figures/chapterX/`

`Config.__init__` checks for this file at startup and halts execution with an error message if it is missing.

### figures/chapterX/

```
chapterX/
├── main.py     ← entry point; loads Config, calls lib/
├── lib/        ← importable modules (no imports from analysis/)
├── plots/      ← generated plots (written by main.py)
└── figures/    ← manual Adobe workspace (not touched by scripts)
```

## Config

`Config` is loaded **once** in `main.py`. Paths and style are passed as arguments to all `lib/` classes — no module inside `lib/` ever imports `config.py` directly.

```python
# In main.py
sys.path.insert(0, str(Path(__file__).resolve().parent / "lib"))
sys.path.insert(0, str(Path(__file__).resolve().parents[2] / "helpers"))
from config import Config

cfg = Config()
# cfg.input_dir  — raw input data
# cfg.output_dir — analysis CSV output root; chapter subdirs live directly here
# cfg.style      — Style dataclass with shared plot settings
```

`Config` reads `config/config.yml`:

```yaml
paths:
  input_dir:  /path/to/data/input
  output_dir: /path/to/data/output/analysis

style:
  dpi: 200
  axis_label_fontsize: 12
  axis_label_fontweight: bold
  tick_fontsize: 8
  tick_fontweight: bold
  kpam_color: "#1f77b4"
  lit_color: "#d62728"
  pyruvylation_color: "#4f81bd"
  acetylation_color: "#f2c94c"
  gray_color: "#bfbfbf"
  both_color: "#DC143C"
```

Analysis scripts write to `cfg.output_dir / "chapterX"`.
Figures scripts read from the same path and write plots to `Path(__file__).resolve().parent / "plots"`.

On startup, `Config` also enforces that every `scripts/analysis/chapterX/` folder (identified by the presence of `main.py`) contains a `README.md`. Execution is halted if any is missing.

## Style

All plot style settings live in `config/config.yml` under `style:`. They are loaded into a `Style` dataclass and accessed as `cfg.style`.

**Passing style to figure scripts:**

```python
# In figures/chapterX/main.py
plot_api = KTypePlotAPI(..., style=cfg.style)
plot_branch_core_heatmap(..., style=cfg.style)
plot_mw_hist(..., style=cfg.style)
```

Figure `lib/` modules accept `style=None` and fall back to hardcoded defaults when called without one. To change a colour or font globally, edit `config/config.yml` — no script changes needed.

The default style follows `year.png` (`plot_cumulative_structures`).

## Imports within lib/

Modules inside `lib/` import each other using plain names (e.g. `from ktypes_base import BaseKTypeAPI`). This works because `main.py` inserts `lib/` into `sys.path` before importing.

## Figure scripts are standalone

`figures/chapterX/lib/` modules have **zero imports from `analysis/`**. All inputs are CSV paths passed as arguments. `figures/chapter2/main.py` can be run any time after `analysis/chapter2/main.py` has produced its CSVs.

## Figure captions

Captions follow the style of the manuscript (Otwinowska et al., *Nature Communications*). Apply these rules whenever writing or editing a figure caption.

### Structure
```
Figure X. Title sentence. (A) Panel description. (B) Panel description.
```

- **Figure X.** — bold, followed by a single bold italic noun phrase that names the overall figure content, ending with a full stop.
- Each panel is introduced with its label in parentheses — **(A)**, **(B)**, **(C)** — followed by one or more descriptive sentences in plain text (not bold).

### Language
- Third person, present tense throughout ("shows", "plot shows", "colours indicate").
- Concise but complete: include key numbers (n=, thresholds, bin widths), colour/symbol meanings, and any methodological choice that affects interpretation.
- No bullet points inside a caption — flowing prose only.
- Species names italicised (*Klebsiella pneumoniae*, *K. pneumoniae*).
- Abbreviations expanded on first use within the caption.

### What each panel description must cover
1. What is shown (plot type and data).
2. What each visual element encodes (axes, colours, symbols, line styles).
3. Any threshold, parameter, or analytical note required to interpret the panel.
4. An optional one-sentence biological interpretation if it adds clarity.

### What to omit
- Do not repeat information that is already conveyed by axis labels or a legend present in the figure.
- Do not speculate beyond what the data show.
- Do not add a data-source sentence unless the figure references an external repository.
