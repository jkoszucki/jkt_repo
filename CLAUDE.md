## processing/gwas-proc — shared data preparation

`processing/gwas-proc/` is a **shared data preparation module** that feeds both chapters 2 and 3. It does not belong to either chapter exclusively. Its job is:

1. Filter pyseer GWAS hits by mode=lasso, pvalue_corr ≤ 0.05, precision ≥ 0.60, F1 ≥ 0.50, MCC ≥ 0.50; write `gwas_hits.tsv`
2. Classify each (clustering_level, PC) by `reported_topology_PC` (joined from `clusters_functions_best_all.tsv`) into four folders: `sgnh-ecod-reported-topology`, `ssrbh-ecod-reported-topology`, `other-ecod-reported-topology`, `no-ecod-reported-topology`
3. Export per-PC data into the 4D output hierarchy: `gwas-data/{ecod-type}/{k-locus}/{clustering-level}/{PC}/`; includes alignment FASTAs, AF3 structures, representative sequences, BLASTP vs prophage hits, K-locus GenBank datasets, and wGRR scores
4. Select best SGNH predictor per K-locus: `gwas_sgnh_hits.tsv` (highest F1 from `sgnh-ecod-reported-topology/`)

`processing/acetyl-proc/` handles chapter 2 specific acetyltransferase detection and the unified table.

`figures/chapter2/` and `figures/chapter3/` are **plotting only** — they read from `gwas-data/` and produce plots. No data computation or file export belongs in figure scripts.

`processing/cps-proc/` feeds figures/chapter4/ in the same way.

## Config

`Config` is loaded **once** in `main.py`. Paths and style are passed as arguments to all `lib/` classes — no module inside `lib/` ever imports `config.py` directly.

```python
# In main.py
sys.path.insert(0, str(Path(__file__).resolve().parent / "lib"))
sys.path.insert(0, str(Path(__file__).resolve().parents[2] / "helpers"))
from config import Config

cfg = Config()
# cfg.input_dir    — raw input data
# cfg.output_dir   — analysis output root; chapter subdirs live directly here
# cfg.style        — Style dataclass with shared plot settings
```

All plot style settings live in `config/config.yml` under `style:`. They are loaded into a `Style` dataclass and accessed as `cfg.style`.

## Naming convention — plots and scripts

Plot files and the `lib/` scripts that produce them follow the same naming scheme, derived from the figure and panel they belong to.

**Plot files** (`plots/`):

```
figure1-panelA.png
figure1-panelA.pdf
figure2-panelB.png
```

Both PNG and PDF are written for each panel unless the panel is a multi-file output (e.g. per-locus phandango PNGs which get their own subfolder).

**Figure lib scripts** (`figures/chapterX/lib/`):

```
figure1-panelA.py
figure2-panelB.py
figure3.py          ← single-panel figure; no letter suffix needed
```

Scripts that serve a single figure panel are named `figureN-panelX.py`. Scripts that produce multiple related panels (e.g. all panels of one figure) may use `figureN.py`. Scripts shared across panels or used for data export rather than plotting (e.g. `phandango_export.py`) keep a descriptive name.

The public entry-point function inside the script should match: `plot_figure1_panelA(...)`, `plot_figure2_panelB(...)`, etc.

## Run flags

Both `processing/*/main.py` and `figures/chapterX/main.py` may contain a **run flags block** at the top. These are plain Python booleans that toggle expensive or optional steps without touching `config.yml`.

```python
# ---------------------------------------------------------------------------
# Run flags — toggle steps without modifying config.yml
# When ON: step runs with per-PC checkpointing — skips any PC whose output
# file already exists. Delete the output file to force recomputation.
# When OFF: step is skipped entirely.
# ---------------------------------------------------------------------------
RUN_PROPHAGE_BLAST      = True
PLOT_ALIGNMENT_PROFILES = True
```

Convention:
- **ON** (`True`) → step runs; for expensive per-PC steps, individual PCs are skipped if their output file already exists (checkpoint). Delete the file to recompute.
- **OFF** (`False`) → step is skipped entirely; downstream steps that depend only on the output files still run using whatever files already exist.
- Flags live in `main.py` only — never in `lib/` modules or `config.yml`.

## AlphaFold3 conventions

- Always use **`model_0`** when organising AF3 outputs from raw predictions.
- **SGNH hydrolases** and **SSRBH** proteins are run as **trimers**; all other proteins as **monomers**.
- JSON upload files are prepared as JSON arrays of up to 30 job dicts, one file per batch. GWAS batches use `scripts/helpers/af3_json_prep.py` (`prepare_af3_batches()`) → `output_dir/manual-upload/gwas_batch_NNN.json`. Enzyme batches use `lib/enzyme_export.py` (`prepare_af3_json()`) → `output_dir/manual-upload/enzymes_batch_NNN.json`. AF3 server daily limit is 30 predictions.
- **Duplicate PC names across clustering levels** (e.g. `PC001` at `PCI50C50` and `PCI80C50`) are disambiguated by an internal ID `{clustering_level}_{PC}` (e.g. `pci50c50_pc001`). This ID is used as the AF3 job name and stored in two companion files:
  - `af3_manifest.tsv` — `af3_id | clustering_level | PC | run_as | batch | sequence`
  - `af3_placement.tsv` — `af3_id | locus | ecod_type | clustering_level | PC` (one row per placement; read by the organise/symlink step after download)
- Manual outputs (AlphaFold3, Foldseek, HHphred) are placed under `{output_dir}/manual-outputs/{method}/` — one subfolder per method (`alphafold3/`, `foldseek/`, `hhphred/`). Scripts read from there and organise into the data structure using symlinks where possible.
- Colour structures by **pLDDT (b-factor)**.
- The structure-rendering script handles both monomers and trimers. It has a `RENDER_MODE` flag with two values: `"fast"` (default) and `"accurate"`.

## Figure captions

Captions follow the style of the manuscript (Otwinowska et al., *PLOS Biology*). Apply these rules whenever writing or editing a figure caption.

### Structure
```
Figure X. Title sentence. (A) Panel description. (B) Panel description.
```

- **Figure X.** — bold, followed by a single bold italic noun phrase that names the overall figure content, ending with a full stop.
- Each panel is introduced with its label in parentheses — **(A)**, **(B)**, **(C)** — followed by one or more descriptive sentences in plain text (not bold).

### Language
- Third person, present tense throughout ("shows", "plot shows", "colours indicate").
- Concise but complete: include key numbers (n=, thresholds, bin widths), colour/symbol meanings, and any methodological choice that affects interpretation.
- No bullet points inside a caption — flowing prose only.
- Species names italicised (*Klebsiella pneumoniae*, *K. pneumoniae*).
- Abbreviations expanded on first use within the caption.

### What each panel description must cover
1. What is shown (plot type and data).
2. What each visual element encodes (axes, colours, symbols, line styles).
3. Any threshold, parameter, or analytical note required to interpret the panel.
4. An optional one-sentence biological interpretation if it adds clarity.

### What to omit
- Do not repeat information that is already conveyed by axis labels or a legend present in the figure.
- Do not speculate beyond what the data show.
- Do not add a data-source sentence unless the figure references an external repository.
